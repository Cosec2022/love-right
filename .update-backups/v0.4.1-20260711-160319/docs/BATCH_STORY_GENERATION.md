# 批量生成故事工作流

## 目标

批量生产的对象不是网页，而是符合 Love Right Schema 的内容包：

```text
story.json
results.json
```

所有故事共享同一个运行时。

## 标准流程

### 1. 建立草稿

```bash
npm run new-story -- office-secret "公司里最不能喜欢的人"
```

### 2. 先写 Story Brief

至少确定：

```text
关系母题
主角身份
主要对象
用户真正的两难
主要路线
关键拒绝点
终局数量
主要测量维度
```

### 3. 生成场景图

先生成场景 ID、slot、跳转和 flags，不要先写长文案。

```text
s01 相遇
├── 接受 → s02_together
└── 拒绝 → s02_separate
                ├── 再打开 → s03_reopen
                └── 继续拒绝 → s03_new_route
```

### 4. 填写剧情和选择

每个选择必须同时检查：

```text
是否符合用户能想到的真实反应
是否与下一幕连续
是否改变状态或评分
拒绝是否被尊重
人物是否记得之前发生的事
```

### 5. 生成结果规则

结果包不要写成固定的八份长报告。应拆成可组合模块：

```text
原型基础文案
终局文案
高维度修饰
低维度修饰
过去模式规则组
未来场景规则组
适配对象追加规则
风险提醒优先规则
```

### 6. 自动检查

```bash
npm run check
```

只有检查通过，才允许把目录项状态改成 `published`。

## 给生成模型的核心约束

可将下面内容作为每次生成任务的固定前缀：

```text
你正在为 Love Right 互动叙事引擎生成内容包。
必须输出符合 story.schema.json 与 results.schema.json 的 JSON。
故事必须有真实分支；拒绝、退出和质疑必须改变后续剧情。
共享 slot 的分支场景测量同一类心理取舍，但文案必须符合此前事件。
每个选择只影响 2—4 个底层维度，权重通常在 -2 到 +2。
所有路线必须可达结果，不能存在断路、死循环和不可达场景。
结果使用可能性语言，并引用选择证据；不得冒充心理诊断或真实预言。
先输出场景图摘要，再输出 story.json，最后输出 results.json。
```

## 批量质量门槛

一篇故事满足以下条件才发布：

- 自动校验全部通过
- 人工完整走通至少四条路线
- 每个终局至少可达一次
- 拒绝路线没有被强行拉回原人物
- 选项没有明显正确答案
- 结果之间有实质差异
- 手机端完整体验无断裂
