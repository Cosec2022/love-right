# Love Right 16维空间评分模型

## 1. 不再只保存一个总分

用户完成18幕以后，系统保存的不是单个类型分数，而是：

```text
A[T][N]  每次答案的空间向量
C[K][N]  不同关系情境下的画像
P[T][N]  从第一幕到第十八幕的心理轨迹
V[N]     同一维度的摇摆程度
D[N]     前半段到后半段的变化趋势
G[N][N]  维度共同出现与显式交互
```

当前：

```text
T = 18 个测量槽位
N = 16 个恋爱轴
K = 6 类关系情境
```

## 2. 16个双向轴

### 靠近与依恋

| ID | 低端 | 高端 |
|---|---|---|
| `approach` | 保护性退后 | 主动靠近 |
| `trust` | 先验证 | 先相信 |
| `certainty` | 容忍模糊 | 需要说清 |
| `abandonment` | 不易被牵动 | 容易担心失去 |

### 浪漫感知

| ID | 低端 | 高端 |
|---|---|---|
| `idealization` | 现实验证 | 赋予浪漫意义 |
| `novelty` | 熟悉稳定 | 未知吸引 |
| `discernment` | 凭感觉 | 看行动验证 |
| `jealousy` | 不易比较 | 对第三人敏感 |

### 沟通与冲突

| ID | 低端 | 高端 |
|---|---|---|
| `expression` | 收住不说 | 直接表达 |
| `conflict` | 回避冷却 | 当场处理 |
| `repair` | 切断止损 | 解释修复 |
| `reassurance` | 内在消化 | 需要回应安抚 |

### 长期关系

| ID | 低端 | 高端 |
|---|---|---|
| `autonomy` | 关系融合 | 保持独立 |
| `commitment` | 重视当下 | 面向长期 |
| `care` | 更愿接受照顾 | 更习惯照顾别人 |
| `validation` | 内在自足 | 需要被明确选择 |

轴是连续方向，不是道德高低。`-0.8` 与 `+0.8` 都可能是合理答案，只表示不同关系策略。

## 3. 单个选择的数据

```json
{
  "context": "ambiguity",
  "intensity": 1.0,
  "confidence": 0.95,
  "vector": {
    "certainty": 0.72,
    "validation": 0.48,
    "autonomy": -0.20
  },
  "cross": [
    {"axes": ["certainty", "validation"], "value": 0.45}
  ]
}
```

- `vector`：这一答案在各轴上的方向，范围 `[-1, 1]`
- `context`：答案发生在哪种关系压力下
- `intensity`：这一幕的心理强度
- `confidence`：该答案作为测量证据的可信程度
- `cross`：某两个轴同时出现时的额外关系

内容文件使用带名字的对象，Score Engine 内部编译为 `Float64Array(16)`，避免数组下标在以后扩维时错位。

## 4. 画像计算

每个轴使用加权平均，并加入轻量先验，避免少量答案把结果推到伪精确极值：

```text
profile[n] = Σ(vector[n] × intensity × confidence)
             ─────────────────────────────────────
             Σ(exposure weight) + prior weight
```

页面显示时映射到8—92：

```text
显示分 = 50 + 42 × profile[n]
```

空间内部仍然保留 `[-1, 1]` 的连续值。

## 5. 结果原型

当前8种叙事原型：

- 清醒慢热型
- 稳定确认型
- 浪漫共振型
- 主动直球型
- 自由靠近型
- 偏爱确认型
- 温柔修复型
- 反差拉扯型

原型不是把用户固定分类，而是结果报告的叙事入口。最终报告还会结合：

```text
终局选择
+ 16维空间画像
+ 情境画像
+ 维度交互
+ 前后趋势
+ 关键选择证据
+ 隐含的对象魅力偏好
```

## 6. 匹配与混合结果

当前原型评分综合考虑方向相似、空间距离、情境表现、维度组合和稳定性。结果先排序，再判断是否输出混合型。

v0.4.1 的混合规则：

- 第一、第二名差距必须小于 `0.0065`
- 两种原型必须在 `blendPairs` 中被声明为合理组合
- 不再因为原型中心距离普遍接近，就让八成用户都得到“带一点另一型”

## 7. 发布校准

`npm run audit` 默认每篇故事运行16,000条随机合法路线，并检查：

- 任一主结果占比不得超过30%
- 每个正式原型占比至少5%
- 不得存在0%死结果
- 混合结果比例保持在12%—55%
- 全A、全B、全C、全D至少形成3种主结果

随机测试不是心理学常模，只是用于发现工程上的塌缩、死结果和错误权重。

## v0.4.2 展示边界

16项底层轴与8项派生 meter 仍会计算，但 meter 不再作为结果首页组件。它们只用于：

- 生成四个生活化关系动作；
- 选择详细解读规则；
- 调试与研究模式；
- 跨故事画像的未来输入。

用户的主结果由记忆标题、核心矛盾和三个结论构成。不要通过增加更多进度条来解释模型。
