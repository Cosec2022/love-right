# 批量生成故事工作流

## 批量生产的对象

不是复制网页，而是生成符合统一契约的内容包：

```text
story.json
results.json
```

所有故事共享 Story / Score / Result Engine。

## 1. 创建16维草稿

```bash
npm run new-story -- office-secret "公司里最不能喜欢的人"
```

脚手架会继承：

- 16维恋爱空间
- 6类关系情境
- 8个页面指标
- 8种结果原型词汇
- v2 Schema

脚手架只是3幕结构样例，正式发布前必须扩展为18个 assessment slots。

## 2. Story Brief

先定义：

```text
目标用户与叙事视角
关系母题
对象的基础魅力与缺点
用户真正的两难
拒绝点和退出路线
现实压力
4个终局
本篇重点覆盖的空间轴
```

对象不能只靠外貌和权力制造魅力。至少需要：

```text
一种稳定能力
一种独特表达方式
一种对边界的尊重
一个真实限制或缺点
```

## 3. 先画场景图

```text
s01 相遇
├── 接受 → s02_together
└── 拒绝 → s02_separate
                ├── 再打开 → s03_reopen
                └── 继续拒绝 → s03_new_route
```

先保证分支逻辑，再写长文案。

## 4. 设计18个空间取舍

每个 slot 的选项应形成方向差异。例如：

```text
主动靠近  ↔  保护性退后
直接表达  ↔  收住观察
需要说清  ↔  容忍模糊
共同修复  ↔  切断止损
```

单个答案通常影响3—6个轴，使用 `[-1,1]` 向量。不要让全部答案只产生正值。

## 5. 隐含对象魅力

在前3—4幕通过用户注意到的细节记录 `appeal`：

```text
steady / witty / gentle / direct
```

同一偏好在第6、12、17幕自然回响，但不改变用户已经作出的事实选择。详见 `LEAD_APPEAL_SYSTEM.md`。

## 6. 结果规则

结果不是固定八篇报告，而是组合：

```text
主原型 / 可选混合型
+ 终局
+ 高低空间轴
+ 情境差异
+ 过去模式
+ 未来三幕
+ 对象魅力偏好
+ 风险提醒
+ 关键答案证据
```

## 7. 发布检查

```bash
npm run check
```

只有以下全部通过，才把 `status` 改为 `published`：

- 所有路径18个答案完成
- 拒绝和退出得到尊重
- 无断路、循环和不可达场景
- 返回上一幕正确回滚
- 8种主结果全部可达
- 单一结果占比不超过30%
- 混合结果不过量
- A/B/C/D固定路线至少产生3种主结果
- 人工走通至少4条路线

## 生成模型固定约束

```text
你正在为 Love Right v2 空间引擎生成互动恋爱故事。
必须输出 story.json 与 results.json。
正式故事固定18个 assessment slots，每幕2—4个真实取舍。
每个选择使用具名16维 vector，值在 -1 到 1；不得全部单向加分。
拒绝、退出、质疑、确认关系和现实选择必须改变后续人物行为。
对象魅力需要通过行动和细节呈现，不得直接宣布“他就是你的类型”。
结果只能使用可能性语言，不得声称心理诊断、读取真实经历或确定预言。
所有路线必须通过 npm run check。
```
